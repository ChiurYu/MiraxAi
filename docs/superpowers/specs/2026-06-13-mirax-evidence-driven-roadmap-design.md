# Mirax AI 证据驱动完整复刻路线图设计

## 目标

Mirax AI 的下一阶段总目标是把旧版「轻语 IP 智能体」完整拆解为一套可验证、可派工、可重建的产品路线。

旧版的页面入口、业务能力、表单字段、状态变化、产物链路和模块边界都要尽量完整复刻。旧版 UI 不作为视觉目标，新版 Mirax AI 会重新设计一套更专业、清晰、适合长期生产使用的界面。

这份路线图不直接进入编码。它先建立旧版功能知识库，再把旧版功能映射到新版产品架构，最后转换成可由 Codex、Claude Code 或 cmux 工位执行的 implementation plans。

## 范围

本路线图覆盖：

- 旧版可运行应用中的所有可见页面、导航、表单、按钮、弹窗、状态和提示。
- 旧版安装包、解包资源、preload API、本地配置、资源目录和旧仓库文档中的功能线索。
- 登录、激活、云服务、模型调用、平台发布等受限能力的运行障碍记录。
- 旧版功能到新版 Mirax AI 信息架构、Provider、sidecar、本地数据和发布链路的映射。
- 后续可派工任务卡需要的文件范围、测试方式、验收标准和优先级。

本路线图不覆盖：

- 绕过登录、激活、会员、权限或付费限制。
- 复用旧版混淆代码或生产代码。
- 像素级复刻旧版视觉设计。
- 在路线图阶段直接实现具体功能。

## 核心原则

1. 先完整盘点，再系统实现。
2. 复刻功能、流程、状态和产物链路，不机械照搬旧 UI。
3. 运行态证据优先，静态分析补盲区。
4. 每条功能都要有证据等级、可信度和来源。
5. 每个后续任务都要能被 Claude Code 独立理解和执行。
6. 每个阶段完成后都要更新项目状态入口，避免新会话丢失上下文。

## 证据模型

每个旧版功能都使用证据等级记录，避免把亲眼看到的页面、静态分析线索和推测混在一起。

| 等级 | 名称 | 定义 |
| --- | --- | --- |
| E1 | 运行态已验证 | 旧版应用中实际打开页面、看到控件、操作过交互或看到状态变化。 |
| E2 | 运行态可见但受限 | 页面、按钮、字段可见，但登录、激活、云服务、模型能力等导致不能完成动作。 |
| E3 | 静态分析确认 | 来自 DMG、ASAR、preload API、资源目录、本地数据库、配置文件等，能确认模块、接口、字段或资源存在。 |
| E4 | 旧仓库或文档线索 | 来自旧 README、早期源码、演示视频、注释、历史文档。 |
| E5 | 合理推断 | 基于页面命名、接口名、模块边界或同类产品逻辑推断，必须标注不确定点。 |

证据来源优先级为：旧版运行态观察，其次旧包和解包内容，再其次旧仓库、文档和演示视频。演示视频覆盖不足，只作为低覆盖参考，不作为完整功能边界。

每条证据还必须标注可信度：

- `high`：证据直接、可复查，且没有相互冲突的来源。
- `medium`：证据可信但不完整，或需要其他来源补足输入输出、状态、依赖。
- `low`：来自推断、过期文档、受限页面或无法复查的观察，只能作为待确认线索。

证据冲突时，优先采用更接近真实运行行为的来源：E1 优先于 E2，E2 优先于 E3，E3 优先于 E4，E4 优先于 E5。同等级冲突时，以时间更新、可复查性更强、信息更完整的证据为准，并在功能卡中保留冲突说明。

### 证据 ID

所有证据都在 `docs/reverse-engineering/evidence-index.md` 中登记，并使用统一 ID。功能卡、页面卡、运行障碍记录和静态分析记录只引用证据 ID，不重复粘贴长路径或大段说明。

证据 ID 格式：

- `EV-RUNTIME-001`：运行态观察、截图、录屏、页面操作记录。
- `EV-STATIC-001`：旧包、解包资源、preload API、本地配置、数据库或代码静态分析。
- `EV-DOC-001`：旧仓库、README、历史文档、演示视频时间轴。
- `EV-INFER-001`：明确标注为推断的分析结论。

证据索引至少记录：证据 ID、证据等级、可信度、来源类型、摘要、资产路径或外部位置、关联页面、关联功能卡、记录人、记录日期。

## 功能卡

每个旧版功能最终都要落成一张功能卡。功能卡是后续设计、计划和派工的最小产品单元。

功能卡至少包含：

- 功能名称。
- 旧版入口。
- 新版归属模块。
- 证据等级。
- 可信度。
- 证据 ID 列表。
- 用户目标。
- 页面结构，包括关键区域、按钮、表单字段、弹窗和状态。
- 输入输出，包括用户输入、系统产物和下游依赖。
- 执行链路，包括前端操作、Provider、sidecar、本地存储和外部服务。
- 限制与风险，包括登录限制、云服务不可用、模型不可用、平台规则变化。
- 新版实现建议，包括保留、重做、合并、延期或替代实现。
- 派工信息，包括建议文件范围、测试方式、验收标准和优先级。

访问限制单独进入运行障碍记录，但不成为绕过任务。运行障碍只说明什么被挡住、在哪里被挡住、能看到什么、静态线索能补充什么，以及新版应如何自研替代。

- 执行链路。
- 依赖模块或外部服务。
- 限制与风险。
- 新版实现建议。
- 优先级。

- 页面名称。
- 旧版入口路径。
- 关联证据 ID 列表。
- 最高证据等级。
- 可信度。
- 页面区域。
- 可见控件。

- 能补强的证据等级。
- 不确定点。
- 新版设计影响。

## 阶段路线

### 阶段 0：整理现状与盘点工具

统一目录和模板，包括功能卡模板、页面巡检模板、运行障碍记录模板、证据索引、截图和录屏命名规则、旧包静态分析记录格式。

把现有 `demo-video-timeline`、`demo-video-coverage`、`legacy-ui-gap-list` 和旧包分析结论纳入同一套证据模型。

阶段 0 摄入的现有文档包括：

- `docs/reverse-engineering/demo-video-timeline.md`
- `docs/reverse-engineering/demo-video-coverage.md`
- `docs/reverse-engineering/legacy-ui-gap-list.md`
- 旧包分析结论。如结论分散在本地文件、聊天记录或临时目录中，阶段 0 先登记当前位置和摘要，再决定是否汇总到 `docs/reverse-engineering/static-analysis/`。

阶段 0 还应产出写入路径分配约定：

- `docs/reverse-engineering/evidence-index.md` 由 Codex 总控或指定工位维护。
- `docs/reverse-engineering/function-cards/` 按模块或页面分配给具体工位。
- `docs/reverse-engineering/pages/` 按导航入口分配给具体工位。
- `docs/reverse-engineering/static-analysis/` 按分析对象分配给具体工位。
- `docs/superpowers/PROJECT-STATE.md` 由 Codex 总控统一更新。

阶段 0 的完成标准：

- 新建或更新项目状态入口。
- 明确证据索引和功能卡目录。
- 明确页面巡检、运行障碍和静态分析记录格式。
- 明确新会话恢复步骤。
- 更新 `PROJECT-STATE.md`，把下一步指向阶段 1 巡检计划。
- 产出阶段 1：旧版运行态全量巡检 implementation plan。

### 阶段 1：旧版运行态全量巡检

使用能打开但受限的旧版应用逐页巡检。按导航和页面入口记录所有页面、弹窗、表单、按钮、状态、提示和可见产物。

阶段 1 巡检按优先级推进：

- P0： 首页工作台、发布流程、设置。这些页面直接影响核心工作流。
- P1：声音管理、形象管理、素材管理、任务中心、账号管理。
- P2：帮助、关于、弹窗提示、边缘状态和低频入口。

阶段 1 plan 应按 P0 → P1 → P2 巡检。无法执行的动作标为 E2，不阻塞继续巡检。每个页面至少产出页面卡；功能卡只为影响新版实现决策的功能拆出，不为每个按钮机械建卡。

阶段 1 的停止条件：

- 所有已巡检页面卡达到最小字段后，阶段 1 才可结束；P0 页面卡必须全部完成，才能进入阶段 2。
- 功能卡是否拆出，由该能力是否影响新版信息架构、Provider、sidecar、本地数据或发布链路决定。
- 低价值按钮、重复入口或纯提示文案可以留在页面卡中，不单独拆成功能卡。

阶段 1 的完成标准：

- 能回答旧版有哪些用户可见页面和能力。
- 每个导航入口都有页面巡检记录。
- 每个影响新版实现决策的可见功能都有初版功能卡。
- 每个受限功能都有运行障碍记录。

### 阶段 2：旧包静态分析补盲区

针对阶段 1 中的空白、受限和不确定功能，分析 DMG、解包资源、preload API、本地配置、旧仓库线索。

重点不是复用旧代码，而是确认模块名、接口名、字段、服务依赖和数据流。

静态补证也按 P0 → P1 → P2 推进，优先补核心工作流。对于只证明“运行态受限、新版需自研替代”的 E2 功能，如果其旧版依赖登录、激活、云服务或已不可用第三方服务，且不会影响新版模块边界，可以记录为运行障碍和替代实现建议，不必深入解包。

阶段 2 的完成标准：

- E2、E4、E5 功能必须完成一次静态补证；E3 如果来源不完整、字段不足或影响新版实现决策，也要继续补证。
- 能升级证据等级的功能已更新功能卡。
- 不能升级的功能明确保留为待确认，并说明缺口。
- 静态分析结果能服务新版 Provider、sidecar 和数据模型设计。

### 阶段 3：Mirax AI 新版产品架构映射

把旧版功能卡映射到新版信息架构和工程模块，包括桌面工作台、素材管理、声音管理、形象管理、任务中心、账号管理、设置、Provider、sidecar、本地数据和发布链路。

UI 重新设计，不复制旧版视觉，但必须覆盖旧版功能意图。

复杂页面、跨模块流程和会显著影响操作效率的功能，应单独使用 UI/UX skill 输出设计方案。简单控件和低风险表单可以直接在功能卡中定义信息结构和行为，不必单独产出 UI 设计文档。

UI/UX skill 的产物应作为模块设计说明或设计决策记录保存，并由相关功能卡引用；功能卡本身保留用户目标、信息结构、行为和验收标准，不承载完整视觉稿。

阶段 3 和阶段 4 可以按模块分批推进，不要求所有功能卡完成 UI 设计后才生成 implementation plans。

阶段 3 的完成标准：

- 形成新版模块地图。
- 每个旧版功能都有新版归属。
- 每个功能标注保留、重做、合并、延期或替代实现。
- 对复杂页面和跨模块流程，明确是否需要 UI/UX skill 设计产物及其存放位置。
- 能按用户流程和工程模块两种视角排序。

新版允许偏离旧版功能的情况必须记录决策理由：

- 旧版功能明显临时、冗余、重复或已废弃时，可以标注为合并或剔除。
- 旧版功能依赖已不可用服务、旧平台规则或不可复用云能力时，可以标注为替代实现。
- 偏离不得绕过登录、激活、会员、权限或付费限制，也不得复用旧版混淆代码或生产代码。

### 阶段 4：可派工实施计划生成

把功能卡转换为 superpowers implementation plans。每个计划包含目标、范围、目标文件、测试命令、验收方式和 gap-list 更新规则。

阶段 4 才把功能卡批量转换为面向实现的 implementation plans，并进入 cmux dispatch 和具体实现。阶段 0 和阶段 1 的盘点计划也通过 `writing-plans` 产出。

阶段 4 的完成标准：

- 每个计划都能被 Claude Code 独立读取和执行。
- 每个计划顶部都有恢复上下文区块。
- 每个计划明确文件范围、验证命令和阶段验收。
- 只有 plan 任务明确对应 `legacy-ui-gap-list.md` 某一行，并且完成标准表示实现或验收已通过时，才更新该行状态。普通逆向盘点、证据发现和功能确认不更新 gap-list 状态。

## 文档结构

路线图后续配套文档应采用以下结构：

```text
docs/
  superpowers/
    PROJECT-STATE.md
    specs/
      2026-06-13-mirax-evidence-driven-roadmap-design.md
    plans/
  reverse-engineering/
    evidence-index.md
    runtime-blockers.md
    function-cards/
    pages/
    static-analysis/
    assets-index.md
```

`docs/superpowers/PROJECT-STATE.md` 是恢复上下文的固定入口。

`docs/reverse-engineering/evidence-index.md` 是证据索引，记录每条证据的判断结果、来源、等级、可信度和对应功能。

`docs/reverse-engineering/function-cards/` 存放旧版功能卡，每个功能一张小文件，方便 Claude Code 独立读取和修改。

`docs/reverse-engineering/pages/` 存放旧版页面巡检记录，按导航和页面命名。

`docs/reverse-engineering/runtime-blockers.md` 记录登录、激活、云服务、模型、平台限制等运行障碍。

`docs/reverse-engineering/static-analysis/` 存放旧包、preload API、资源目录、配置和旧仓库分析记录。

`docs/reverse-engineering/assets-index.md` 只记录截图、录屏、解包结果和其他证据资产的位置、来源、体积、Git 策略和关联证据 ID。

`docs/superpowers/plans/` 存放后续由 `writing-plans` 产出的实施计划，继续遵守项目约定：取最新 `*.md` 作为自动调度入口。每个最新计划的标题和 `Resume Here` 区块都要明确它是当前自动调度入口，避免旧计划和新计划抢入口。阶段计划命名应带阶段号和任务主题，例如 `2026-06-13-stage-0-inventory-tools.md`，与 specs 文件区分。

大型逆向文件不进入 Git，包括 DMG、完整录屏、解包 ASAR、模型文件、大型资源目录和批量抽帧图片。Git 内只保存索引、摘要和必要的小体积截图。若截图需要入 Git，应放在 `docs/reverse-engineering/assets/screenshots/`，按 `YYYY-MM-DD-page-or-feature-brief.png` 命名，并在 `assets-index.md` 和 `evidence-index.md` 同时登记。录屏和大体积解包结果只记录本机路径、外部存储位置或摘要，不直接纳入 Git。

## 恢复上下文与进度指引

每次重新进入聊天或启动 Claude Code 工位时，都必须能在 1 到 3 分钟内知道当前目标、最新进度和下一步动作。

为此需要维护固定入口 `docs/superpowers/PROJECT-STATE.md`。它是项目驾驶舱，不替代详细 spec 或 plan，但负责指向它们。

`PROJECT-STATE.md` 应包含：

- 当前总目标。
- 当前阶段。
- 当前阶段完成标准。
- 最新可执行任务。
- 进度入口，包括路线图 spec、最新 implementation plan、evidence index、function cards、runtime blockers 和 gap-list。
- 最近关键决策。
- 工作区注意事项，包括未提交改动、大型逆向文件不入 Git、CodeGraph、cmux 和 AGENTS 规则。
- 新会话恢复步骤，例如依次读取 `AGENTS.md`、`CLAUDE.md`、`docs/superpowers/PROJECT-STATE.md`、最新 spec 或 plan。
- 下一步命令或动作。

每份后续 implementation plan 顶部也要保留 `Resume Here` 区块，说明当前任务、已完成任务、下一步任务和验证命令。

`.codex/dispatch-state.json` 仍作为 cmux 派工运行态状态，但它不是新会话和人工协作的主入口。

阶段完成时，如果没有更新 `PROJECT-STATE.md`，则阶段不算完成。

## 协作方式

Codex 总控负责路线图、计划、验收和提交。

Claude Code 或 cmux 工位负责按功能卡、页面卡或明确的计划任务执行具体盘点和实现。

每个工位任务必须声明写入路径，避免多人同时修改同一份大文档。功能盘点优先使用小文件，证据索引只记录摘要和跳转。

需要 UI 重新设计的功能，功能卡只定义用户目标、信息结构和行为，不锁死视觉方案。后续可单独使用 UI/UX skill 输出设计方案，再进入实现。

## 风险与处理

旧版运行态受限是主要风险。处理方式是先完整记录可见页面和控件，把执行失败或不可用能力写入 `runtime-blockers.md`，再用静态分析补充证据。限制不阻塞主线，但必须标注可信度。

盘点文档失控是第二个风险。处理方式是总入口清晰、功能卡小文件化、证据索引统一编号，避免所有人同时改一份巨型文档。

完整复刻变成机械照搬是第三个风险。处理方式是明确复刻的是功能、流程、状态和产物链路，不复刻旧版视觉。新版 UI 走重新设计。

多工位方向不一致是第四个风险。处理方式是每张功能卡和每份计划都带文件范围、验收标准、证据来源和下一步动作；阶段完成必须更新 `PROJECT-STATE.md`。

## 验收标准

本路线图设计通过后，应满足：

- 明确长期愿景和近期落地节奏。
- 明确以旧版完整功能复刻为目标，UI 重新设计。
- 明确证据等级 E1 到 E5。
- 明确可信度字段和证据冲突判定规则。
- 明确证据 ID 格式和证据索引职责。
- 明确功能卡、页面卡、运行障碍记录、静态分析记录的用途。
- 明确功能卡、页面巡检卡、运行障碍记录、静态分析记录的最小模板字段。
- 明确阶段 0 到阶段 4。
- 明确 `PROJECT-STATE.md` 是恢复上下文的固定入口。
- 明确后续进入 `writing-plans`，先写阶段 0：盘点工具与项目状态入口的 implementation plan。
- 明确阶段 0 验收时再产出阶段 1 巡检 plan。
- 明确证据资产落点和大型文件不入 Git 的规则。
- 明确普通逆向盘点不更新 `legacy-ui-gap-list.md` 状态。
- 不包含绕过登录、激活、会员、权限或付费限制的目标、步骤或验收标准。
